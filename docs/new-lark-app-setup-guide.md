# ミニえのぽこまん 緊急復旧手順書

## 📢 現状のご報告

本日夕方16時以降、複数アカウントでログインできない事象が発生しています。

**Lark APIの月間呼び出し回数上限に到達**しました。

ミニえのぽこまんの認証・データ保存はすべてLark Base経由で動作しており、
168名の講座メンバーが利用を開始したことで、Lark APIの月間クォータを
本日使い切ってしまった状態です。

```
エラーメッセージ（Lark公式）:
"This month's API call quota has been exceeded"
(今月のAPIコール数上限を超えました)
```

## 🔧 クライアント様作業手順

### STEP 1: Lark Developer Consoleで新アプリ作成

1. Lark Developer Console にアクセス
   → https://open.larksuite.com/app

2. 「カスタムアプリを作成」をクリック

3. 以下の情報を入力:
   - **アプリ名**: `ミニえのぽこまん v2` （任意の名前でOK）
   - **アプリの説明**: `チャットボット用 / クォータ復旧版`
   - **アイコン**: 任意（現行アプリのロゴを流用可）

4. 「作成」をクリック

---

### STEP 2: APP_ID / APP_SECRET を取得

1. 作成したアプリの「認証情報と基本情報」タブを開く

2. 以下の2つをコピー:
   - **App ID**: `cli_xxxxxxxxxxxxxxxx`
   - **App Secret**: `xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`

3. 📧 開発担当（有村）にセキュアな方法で共有
   （※ Lark DMで送付推奨。メール平文は避ける）

---

### STEP 3: アプリに権限を付与

「権限とスコープ」タブで以下の権限をすべて有効化:

#### 🔹 Bitable（多次元表）権限
- 多次元表の表示・管理 (`bitable:app`)
- 多次元表レコードの閲覧 (`bitable:app:readonly`)
- 多次元表レコードの作成・更新 (`bitable:app:write`)

#### 🔹 その他（現行アプリと同じ権限）
現行アプリの権限設定をスクリーンショットで共有いただければ、
同じものを設定できます。

---

### STEP 4: アプリをリリース（有効化）

1. 「バージョン管理とリリース」タブ
2. 「バージョンを作成」→ バージョン番号を入力（例: `1.0.0`）
3. 「リリース」→ 管理者承認を申請
4. 管理者（社内のLark管理者）が承認すると有効化完了

---

### STEP 5: Baseへのアクセス権を追加

1. ミニえのぽこまんで使っている Lark Base を開く
2. 右上の「共有」または「設定」→「権限設定」
3. 「メンバーを追加」で新アプリ名（例: `ミニえのぽこまん v2`）を検索
4. 権限: **編集者** を付与

## 📊 追加で実施済みの改善

クライアント様作業を待つ間に、以下の最適化を本番デプロイ済みです:

### ログインAPIの最適化（API呼び出し数を大幅削減）
- **変更前**: ログイン1回につき全ユーザー168名分を取得
- **変更後**: ログイン1回につき該当ユーザー1名のみ取得
- **効果**: **API呼び出し数が約1/168に削減**

これにより、新アプリ切替後はクォータを大幅に節約でき、
同様の問題が再発するリスクが大幅に下がります。

## ❓ よくある質問

### Q. 既存のアプリは削除しないといけない？
**A.** いいえ、そのまま残してOKです。
5月に月間クォータがリセットされたら、元のアプリに戻す選択も可能です。

### Q. 5月以降も新アプリのまま運用で問題ない？
**A.** はい、問題ありません。
アプリ名が変わるだけで、Baseのデータは同じものを利用します。

### Q. 費用はかかる？
**A.** Lark Developerの新アプリ作成は**無料**です。
追加コストはかかりません。

### Q. 旧アプリのクォータはいつリセット？
**A.** 通常、毎月1日にリセットされます（5月1日復活見込み）。

### Q. なぜ今回こうなった？
**A.** 168名の講座メンバーが利用開始したことで、想定以上の
API呼び出しが発生したためです。今回の最適化（上記）で
今後同様の問題は起こりにくくなります。
